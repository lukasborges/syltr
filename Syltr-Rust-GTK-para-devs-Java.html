O que é o Syltr?

O objeto de estudo — um app desktop real, não um "hello world"

Em uma frase

Junta WhatsApp, Telegram, Slack, Discord & cia. numa única janela nativa do GNOME, cada serviço com sessão isolada (cookies/login separados).

  • Barra lateral de ícones com favicons e badge de não-lidas
  • Notificações nativas, mudo por serviço, "não perturbe"
  • Corretor ortográfico do sistema, downloads, atalhos
  • Links externos abrem no navegador padrão

A stack (e o que cada peça é)

  • Rust — linguagem: nativa, sem GC, foco em segurança de memória
  • GTK4 — toolkit de UI (widgets, janelas, eventos)
  • libadwaita — componentes visuais do GNOME sobre o GTK
  • CEF — Chromium embarcado que renderiza cada web app
  • Cargo — build + gerenciador de dependências (≈ Maven)

JavaRust
JVM + bytecodeBinário nativo, sem VM
Garbage Collector em runtimeOwnership decidido em compilação
null → NullPointerExceptionOption<T> (sem null)
Exceptions (try/catch)Result<T,E> (erro é valor)

struct + impl em vez de classe

Rust separa os DADOS (struct) do COMPORTAMENTO (bloco impl). Exemplo real: o Service do Syltr.
src/config/service.rs
#[derive(Clone, Serialize, Deserialize)]
pub struct Service {
    pub id: String,
    pub name: String,
    pub url: String,
    #[serde(default)]
    pub muted: bool,
}

// métodos ficam num bloco separado:
impl Service {
    fn services_file() -> PathBuf { ... }
}

Para um dev Java

  • struct = os campos (como o estado de uma classe)
  • impl = onde vivem os métodos
  • #[derive(...)] gera código automático — Clone, e (de)serialização JSON via serde
  • Sem herança — reuso é via composição e traits
  • pub campo a campo controla o encapsulamento

Ownership & Borrowing

O conceito que substitui o Garbage Collector
  • Cada valor tem UM dono. Quando o dono sai de escopo, a memória é liberada — automático, sem GC.
  • Você pode EMPRESTAR o valor: &T (leitura, vários) ou &mut T (escrita, um só de cada vez).
Empréstimo de leitura (&)
fn save(list: &[Service]) {  // empresta
    // ... só lê a lista
}

let servicos = config::load();  // dono
save(&servicos);   // empresta p/ save
save(&servicos);   // ainda posso usar!

Por que isso importa

  • Sem vazamento e sem "use after free" — o compilador prova em tempo de build
  • & vs &mut é como o Rust garante que ninguém escreve enquanto outro lê
  • No Java o GC resolve na marra em runtime; aqui é decidido antes de rodar

Option e Result

Fim do null e das exceptions — o erro vira um valor
Exemplos reais do Syltr
// Option<T>: tem valor OU não (sem null)
pub fn host(&self) -> Option<BrowserHost> {
    self.host.borrow().clone()
}

// o chamador é OBRIGADO a tratar os 2 casos
if let Some(host) = slot.host() {
    host.was_resized();
}

// Result<T,E>: Ok(T) ou Err(E)
match serde_json::to_string(list) {
    Ok(json) => fs::write(file, json),
    Err(e)   => eprintln!("falhou: {e}"),
}

O mapa mental

  • Option<T> ≈ Optional<T> do Java, mas usado em TODO lugar em vez de null
  • Result<T,E> ≈ retorno de erro checado — sem stack unwinding
  • Some / None os dois lados de um Option
  • Ok / Err os dois lados de um Result
  • O compilador reclama se você esquecer de tratar um caso — o bug não chega em produção

enum + match

Enums que carregam dados, e um switch que o compilador confere
src/engine/download.rs
// nome do arquivo conforme a extensão
let filename = match ext {
    Some(ext) => format!("{stem} ({n}).{ext}"),
    None      => format!("{stem} ({n})"),
};

// 'let-else': sai cedo se não houver item
let Some(item) = download_item else {
    return;
};

Diferente do switch Java

  • match é EXAUSTIVO: se faltar um caso, não compila
  • Cada braço pode extrair os dados de dentro (Some(ext) → ext)
  • É uma expressão: devolve valor direto para a variável
enum em Rust é mais poderoso que o do Java: cada variante pode carregar dados diferentes. Option e Result são só enums da biblioteca padrão — nada mágico.

Traits ≈ interfaces

O handler de downloads implementa a interface (trait) que o CEF chama nos eventos.
src/engine/download.rs
impl DownloadHandler {
    fn on_before_download(&self, ...,
        suggested_name: Option<&CefString>,
        callback: Option<&mut ...>) -> c_int {
        let name = suggested_name
            .map(|n| n.to_string())
            .unwrap_or_else(|| "download".into());
        let path = unique_path(&dir(), &name);
        cb.cont(Some(&path.into()), 0);
        1
    }
}

Traços de trait

  • trait = contrato de métodos (≈ interface)
  • impl Trait for Tipo = implementa o contrato
  • &self ≈ o this do método
  • Pode ter método padrão (≈ default method)
  • A última expressão sem ";" É o retorno (o 1 no fim)

Closures & callbacks

Funções anônimas — a cola entre a UI e a lógica
src/window/mod.rs
// quando uma linha da lista é selecionada:
let ui_selected = ui.clone();
list.connect_row_selected(move |_, row| {
    if let Some(row) = row {
        ui_selected.show_service_at(
            row.index() as usize);
    }
});

Lendo a closure

  • |args| { corpo } é a sintaxe (≈ lambda -> do Java)
  • move transfere posse das variáveis capturadas para dentro
  • ui.clone() antes: dá uma cópia barata do handle p/ a closure guardar
Comparação Java: connect_row_selected(|...| ...) equivale a list.addListSelectionListener(e -> ...). O move existe porque, sem GC, o Rust precisa saber quem é dono das variáveis que a closure leva consigo.

Rc & RefCell

Compartilhar estado mutável sem GC e sem quebrar as regras
  • A UI tem callbacks que precisam ver e alterar o MESMO estado. Sem GC, isso exige ferramentas explícitas.
src/window/mod.rs
struct Ui {
    // vários donos do mesmo valor:
    dnd:   Rc<Cell<bool>>,
    spell: Rc<RefCell<Vec<String>>>,
    state: Rc<RefCell<State>>,
}

// ler/alterar o conteúdo em runtime:
self.state.borrow_mut().current = Some(id);
  • Rc<T> = contagem de referências: vários donos, liberado quando o último some
  • RefCell<T> = permite mutar por dentro, checando as regras em runtime
  • borrow() / borrow_mut() pegam acesso de leitura/escrita
No Java isso seria "só uma referência a um objeto mutável". O Rust te faz declarar a intenção — e paga com segurança.

O que é GTK4 (e libadwaita)?

Analogias para o mundo Java

  • GTK4 = o toolkit de UI (≈ Swing/JavaFX): janelas, botões, listas, layout, eventos
  • libadwaita = widgets e visual do GNOME sobre o GTK (≈ um "look & feel" + componentes prontos)
  • Escrito em C, exposto ao Rust por bindings — daí os prefixos gtk:: e adw::
  • Cairo = a API de desenho 2D (usada no Syltr p/ pintar a página)

Três ideias centrais do GTK

  • Árvore de widgets — tudo é um widget aninhado (janela ⊃ toolbar ⊃ split ⊃ lista)
  • Sinais — widgets emitem eventos; você conecta callbacks
  • Loop principal — uma thread processa eventos e redesenha (como o EDT)

Widgets & o padrão Builder

Como se cria e monta a UI
src/window/mod.rs
let window = adw::ApplicationWindow::builder()
    .application(app)
    .title("Syltr")
    .default_width(1100)
    .default_height(760)
    .content(&root_toolbar)
    .build();

let split = adw::NavigationSplitView::builder()
    .sidebar(&sidebar_page)   // rail esquerda
    .content(&content_page)   // web view
    .build();

Reconhece?

  • É o mesmo Builder pattern que você já usa no Java (StringBuilder, Lombok @Builder)
  • .build() no fim devolve o widget pronto
  • &widget passa uma referência emprestada — o pai não vira dono exclusivo
  • Widgets se aninham: janela → toolbar → split → páginas → lista/stack

Sinais, callbacks e o loop principal

src/main.rs
let app = adw::Application::builder()
    .application_id(APP_ID).build();

app.connect_startup(|_| load_css());
app.connect_activate(window::build);

let code = app.run();   // entra no loop!
  • connect_<sinal> liga um callback a um evento — igual a addXxxListener
  • app.run() bloqueia e roda o loop de eventos até a janela fechar
⚠ Regra de ouro (igual ao Swing/JavaFX)
Toda a UI roda em UMA thread. Nunca bloqueie o loop principal com trabalho pesado — é o mesmo pecado de travar o Event Dispatch Thread no Swing. Trabalho demorado vai para fora e volta via callback/idle. O CEF do Syltr, inclusive, é "bombeado" pelo loop com engine::start_pump().

CEF em modo offscreen (OSR)

Como cada serviço web aparece dentro da janela GTK
1
Browser sem janela
windowless_rendering_enabled = 1
2
Chromium renderiza
Chromium isolado, com sessão/cache própria por pasta
3
Pixels → callback
O CEF entrega os quadros já pintados (render.rs guarda o buffer)
4
Cairo desenha no GTK
Um gtk::DrawingArea pinta o buffer via set_draw_func
5
Entrada de volta
input:: repassa mouse/teclado do GTK ao browser
Por que offscreen? No Wayland/Linux o CEF não pode embutir uma janela nativa dentro do GTK, então renderiza "fora da tela" e o Syltr desenha o resultado. É a peça mais avançada — não precisa dominá-la para começar.
